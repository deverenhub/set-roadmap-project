## Overview

The SET Interactive Roadmap Platform is a modern, real-time web application designed to replace static Excel spreadsheets and LucidChart diagrams with an AI-powered, collaborative digital platform.

### Key Design Principles

1. **Real-time First** - All changes sync instantly across users
2. **AI-Native** - Natural language interface for all operations
3. **Serverless** - No infrastructure management required
4. **Type-Safe** - Full TypeScript across frontend and database
5. **Zero-Trust** - Row Level Security at the database layer

## Technology Decisions

### Why Supabase?

| Requirement | Supabase Solution | Alternative Considered |
|-------------|-------------------|------------------------|
| Real-time sync | PostgreSQL + Realtime | Firebase (vendor lock-in) |
| Authentication | GoTrue (built-in) | Auth0 (separate service) |
| Authorization | Row Level Security | Application-level (risky) |
| API | Auto-generated REST | Custom API (more work) |
| MCP Support | Official MCP server | Custom implementation |

### Why React Flow?

| Requirement | React Flow Solution | Alternative |
|-------------|---------------------|-------------|
| Node/edge diagrams | Core feature | D3.js (complex) |
| Zoom/pan | Built-in | Manual implementation |
| Custom nodes | React components | SVG manipulation |
| Interactivity | Event handlers | Custom event system |
| Performance | Virtualized | Render all nodes |

### Why TanStack Query?

| Requirement | TanStack Query | Alternative |
|-------------|----------------|-------------|
| Server caching | Automatic | Manual Redux |
| Background refetch | Built-in | Custom intervals |
| Optimistic updates | Mutation hooks | Complex Redux logic |
| DevTools | Included | None |
| Real-time integration | Works with subscriptions | Manual sync |

### Why MCP (Model Context Protocol)?

| Requirement | MCP Solution | Alternative |
|-------------|--------------|-------------|
| Standardized tools | Open specification | Custom function calling |
| Context management | Built-in | Manual context window |
| Future compatibility | Growing ecosystem | Vendor-specific |
| Claude integration | Native support | REST wrapper |

### Row Level Security

```sql
-- Example RLS policies

-- Anyone can read capabilities
CREATE POLICY "capabilities_select" ON capabilities
  FOR SELECT USING (true);

-- Only admins and editors can modify
CREATE POLICY "capabilities_modify" ON capabilities
  FOR ALL USING (
    auth.jwt() ->> 'role' IN ('admin', 'editor')
  );

-- Users can only see their own activity
CREATE POLICY "activity_own" ON activity_log
  FOR SELECT USING (
    user_id = auth.uid()
    OR auth.jwt() ->> 'role' = 'admin'
  );
```

## Scalability

### Current Architecture Limits

| Component | Limit | Notes |
|-----------|-------|-------|
| Concurrent users | ~1,000 | Supabase Pro tier |
| Database size | 8 GB | Supabase Pro tier |
| Real-time connections | 500 | Per project |
| Edge Function invocations | 500K/month | Pro tier |
| Claude API | Rate limited | Per API key |

### Scaling Strategies

1. **Database**: Upgrade Supabase tier, add read replicas
2. **Real-time**: Enable multiplexing, optimize subscriptions
3. **Edge Functions**: Implement caching, optimize cold starts
4. **Frontend**: CDN via Vercel, code splitting, lazy loading
